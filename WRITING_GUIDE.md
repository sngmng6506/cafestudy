# Writing Guide

CafeStudy의 사용자 대상 문구 기준. 버튼, 제목, 빈 상태, 오류, 토스트, 안내문을 추가하거나 수정하기 전에 확인한다.

## 목소리

CafeStudy는 함께 공부하는 소규모 커뮤니티다. 문구는 다음 네 가지 성격을 유지한다.

- **명확하게**: 무엇이 일어났고 무엇을 할 수 있는지 바로 이해된다.
- **간결하게**: 의미 없는 제목, 반복 설명, 불필요한 수식어를 줄인다.
- **친근하게**: 사용자 대상 문장은 기본적으로 자연스러운 해요체를 사용한다.
- **존중하며**: 사용자를 탓하거나 명령하지 않고 선택과 해결 방법을 제시한다.

재미 기능은 장난스럽게 표현할 수 있지만 핵심 기능의 오류·결제·삭제·개인정보 문구는 명확성을 우선한다.

## 기본 원칙

### 1. 다음 결과를 예상할 수 있게 쓴다

버튼과 링크에는 눌렀을 때 일어나는 행동을 적는다.

- `확인` → `모임 참여하기`
- `계속` → `사진 인증하기`
- `관리` → `대표 뱃지 바꾸기`
- `삭제` → 대상이 불명확하면 `뱃지 삭제하기`

이미 문맥상 대상이 명확한 작은 팝업에서는 짧은 라벨을 허용한다.

### 2. 한 문장에는 한 가지 핵심만 담는다

제목이 이미 설명한 내용을 본문에서 반복하지 않는다. 사용자가 판단해야 할 정보가 여러 개면 문장을 나눈다.

### 3. 말하듯 쉽게 쓴다

소리 내어 읽었을 때 자연스러운 표현을 사용한다. 구현·서버·데이터베이스 용어를 사용자 문구에 노출하지 않는다.

- `세션 토큰이 만료되었습니다.` → `로그인이 만료됐어요. 다시 로그인해 주세요.`
- `요청 처리에 실패했습니다.` → 가능한 경우 실제 문제와 해결 방법을 안내
- `외부 크롤링 일정` → `소모임에서 가져온 일정`

### 4. 같은 개념에는 같은 단어를 쓴다

용어를 화면마다 바꾸지 않는다. 다른 단어를 쓸 때는 실제 행동이나 데이터 출처가 다를 때만 허용한다.

### 5. 사용자의 감정을 고려한다

성취 순간에는 결과와 보상을 함께 알려 준다. 오류에서는 사용자를 탓하지 않고 복구 방법을 우선 안내한다.

- `인증이 완료되었습니다.` → `오늘 공부를 인증했어요. 10포인트를 받았어요.`
- `잘못된 사진입니다.` → `사진을 확인하기 어려워요. 밝은 곳에서 다시 찍어 주세요.`

과도한 축하나 감탄사는 반복 사용하지 않는다.

## 문체

- 사용자 대상 문장: 해요체
- 짧은 상태 라벨·탭·제목: 명사형 허용
- 관리자·개발 로그: 이 가이드 적용 대상 아님
- 느낌표: 특별한 성취 순간에만 제한적으로 사용
- 마침표: 한 문장짜리 짧은 버튼·라벨에는 사용하지 않음
- 존댓말을 유지하고 반말·과도한 유행어를 사용하지 않음

## 용어 사전

| 개념 | 기본 용어 | 피하거나 구분할 표현 |
|---|---|---|
| 새 모임 생성 | 모임 만들기 | `모임 개설`과 혼용하지 않음 |
| 내부 모임 참석 | 참여하기 | 외부 서비스의 `신청하기`와 구분 |
| 참여 철회 | 참여 취소하기 | `신청 취소`와 혼용하지 않음 |
| 외부 소모임 이동 | 소모임에서 신청하기 | 외부 서비스임을 문구에 표시 |
| 출석 증명 | 출석 인증 | 화면 맥락상 촬영 행동이면 `사진 인증하기` 가능 |
| 앞으로 열릴 일정 | 예정 모임 | `열린 모임`은 모집 상태 의미로만 사용 |
| 완료된 일정 | 지난 모임 | 데이터·메뉴 이름이 이미 `모임 이력`이면 점진적으로 통일 |
| 대표 이미지형 보상 | 대표 뱃지 | `적용`, `설정`, `변경`을 행동에 맞게 사용 |
| 누적 보상 단위 | 포인트 | 점수와 혼용하지 않음 |

새 용어가 여러 화면에 등장하면 이 표에 먼저 추가한다.

## UI 요소별 작성법

### 페이지·섹션 제목

- 내용의 범위를 바로 알 수 있는 짧은 명사형을 사용한다.
- `안내`, `정보`, `관리`처럼 범위가 모호한 제목은 구체화한다.
- 제목과 첫 문장이 같은 내용을 반복하지 않게 한다.

### 버튼

- 동사형으로 작성하고 실제 결과와 일치시킨다.
- 위험 행동은 대상을 포함한다: `모임 취소하기`, `뱃지 삭제하기`
- 뒤로가기, 닫기처럼 관습이 명확한 아이콘 버튼은 `aria-label`에 동작을 쓴다.
- 외부 이동에는 서비스 이름을 포함한다.

### 빈 상태

다음 순서로 작성한다.

1. 현재 상태
2. 가능한 다음 행동

예:

```text
예정된 모임이 없어요.
새 모임을 만들어 보세요.
```

단순히 `데이터가 없습니다.`라고 쓰지 않는다.

### 오류

다음 순서로 작성한다.

1. 사용자 관점에서 무슨 일이 생겼는지
2. 지금 할 수 있는 해결 방법

- 내부 오류 코드와 HTTP 상태를 그대로 노출하지 않는다.
- 사용자가 해결할 수 없는 문제라면 재시도 또는 문의 방법을 안내한다.
- 사용자의 입력이 문제여도 비난하는 표현을 쓰지 않는다.

### 성공·토스트

- 완료된 행동을 과거형으로 명확히 알린다.
- 포인트, 랭킹, 뱃지 등 추가 결과가 있으면 함께 표시한다.
- 화면에서 이미 결과가 명확하면 중복 토스트를 띄우지 않는다.

### 확인 다이얼로그

- 제목에 위험 행동과 대상을 명시한다.
- 본문에는 되돌릴 수 있는지와 영향 범위를 설명한다.
- 버튼은 `취소`와 실제 행동으로 구성한다.

예:

```text
이 모임을 취소할까요?
참여자에게서 일정이 사라지고 되돌릴 수 없어요.
[돌아가기] [모임 취소하기]
```

## 금지하거나 줄이는 표현

- `정상적으로`, `성공적으로`: 성공 메시지에서는 대개 불필요
- `해당`, `관련`, `진행`: 대상을 직접 쓸 수 있으면 제거
- `유효하지 않은`, `실패했습니다`: 구체적인 문제를 설명할 수 있으면 대체
- `확인`, `계속`, `완료`: 버튼 결과가 모호하면 사용하지 않음
- 개발 용어: API, 토큰, DB, 크롤링, payload 등

## 변경 체크리스트

사용자 문구를 추가하거나 수정할 때 확인한다.

- 문구만 보고 다음 행동이나 결과를 예상할 수 있는가
- 같은 개념에 기존 용어를 사용했는가
- 해요체와 명사형이 역할에 맞게 쓰였는가
- 제목과 설명이 같은 내용을 반복하지 않는가
- 빈 상태와 오류가 다음 행동을 안내하는가
- 사용자를 탓하거나 불필요하게 불안하게 만들지 않는가
- 외부 서비스 이동과 위험 행동이 명확히 표시되는가
